## Vision

A Melbourne-Australian small-business website that does three things at once:

1. **Sells two real services** — Data & AI consulting (B2B), VEU energy upgrades
   (B2C, Victorian homeowners under "Unwindai GoGreen")
2. **Showcases free tools** — bank statement summariser, invoice/receipt matcher,
   SMSF eligibility checker — that drive top-of-funnel traffic
3. **Acts as a live AI showroom** — every visitor interaction is itself an example
   of what UnwindAI can build. The chat agent, form intake, and (later) voice
   outreach all double as portfolio pieces

Stack is intentionally simple: Vercel + static HTML/CSS/JS + Anthropic API +
DuckDB + Resend. No frontend framework. No CMS. No build pipeline beyond what
Vercel auto-runs. This is a deliberate Karpathy-style choice — each piece
should be independently understandable in under 10 minutes.

## Phase plan (where we're going)

Phases are designed so each one delivers standalone value. Skip a phase, the
next phase still works on the previous foundation.

### Phase 1: Functionality verification & gadgets — **ACTIVE**

Tactical sprint already captured in `WORKPLAN.md`. Summary:

1. Functionality check across all 4 pages
2. Build out `/gadgets` — three working tools, dispatched as parallel sub-agents:
   - Bank Statement Summariser (PDF/CSV → categorised transactions + monthly summary)
   - Invoice & Receipt Matcher (image/PDF → extracted fields → match by vendor/amount/date)
   - SMSF Eligibility Checker (questionnaire → decision tree → PDF export)

Done when each gadget has a working page, an API endpoint where needed, and
real (not placeholder) functionality.

---

### Phase 2: Form intelligence (current → AI-routed intake)

**Why:** the email-only flow loses signal. AI can categorise, prioritise, and
write to Sheets/DuckDB without you doing manual data entry on every lead.

**Architecture:** `/api/intake` becomes an AI router. Forms POST here in
parallel with the existing email endpoint (email = always-on safety net).
The intake agent:

1. Reads the form payload as a structured intake message
2. Calls tools: `categorise_lead`, `save_to_sheet`, `save_to_duckdb`, `draft_reply`
3. Returns success — the email-fallback already fired, so worst case is no
   AI enrichment, never a lost lead

**Tools needed:**
- `save_to_sheet(rows)` — append to a Google Sheet via service account
- `save_to_duckdb(table, row)` — insert into a local DuckDB (or syncable one)
- `categorise_lead({hot|warm|cold, reason, suggested_response_window})` — pure metadata
- `draft_reply(tone, points)` — returns suggested reply text (saved alongside the lead, not auto-sent)

**Tradeoffs to remember:**
- Cost: $0.01-0.05 per submission. Fine at any reasonable volume
- Latency: 2-5s — fire-and-forget from the form, never block the user
- Prompt injection risk: form input goes inside `<form_data>` tags, hardcode
  destinations in tool implementations (model picks *what* to do, never *where*)

**Done when:** form submission appears in your Sheet within 10 seconds with
correct categorisation, and you have an audit log of every AI decision.

---

### Phase 3: AI-drafted reply emails

Building on Phase 2's `draft_reply` tool:

- After AI categorises a lead, it drafts a personalised reply in your tone
- Reply goes to **you** for review (not auto-sent)
- One-click "Send as-is" or edit & send
- Over time, the system learns your edit patterns — fewer edits needed

**Where it lives:** lightweight admin page at `/admin/leads` (password-protected
or magic-link auth via Resend). Shows pending leads, drafted replies, send button.

**Tone bootstrapping:** seed system prompt with 5-10 of your real past replies
to similar enquiries. Lock writing style: concise, direct, no
corporate-speak, no "I hope this email finds you well".

**Done when:** you can clear an inbox of 10 enquiries in <5 minutes by
reviewing AI drafts, vs typing each one.

---

### Phase 4: AI books a call slot (low-compliance bridge to voice)

Before going full voice, intermediate step: AI emails the lead a Calendly /
Cal.com link with a 1-paragraph brief tailored to their enquiry, plus tells
**you** what to prep before the call.

**Why this step exists:** voice agent is high-risk on first impression. Booking
a call where YOU show up but armed with AI-generated prep is 80% of the value
at 10% of the risk.

**Components:**
- `book_callback` tool: chooses next available slot, sends invite
- AI brief: "Lisa from Frankston, 3-bed, gas hot water, mentioned solar — likely
  candidate for heat pump HW + 2-head split. Probable co-pay $500-1,200. Prep:
  bring system options A and B."
- Outcome capture: after the call, you tag the lead (won / lost / quoted /
  no-show) so the model learns

**Done when:** you can go from "form submitted" to "Lisa booked Tuesday 2pm
with a brief in your inbox" with zero manual steps from you.

### Phase 5: AI voice outreach — Unwindai GoGreen voice agent

**This is the bit you asked about.** Only worth building once Phases 2-4 prove
the lead funnel is healthy and you have call transcripts/outcomes to learn from.

**Build approach:** start with **Vapi.ai** (managed orchestration) so you don't
spend 6 weeks gluing Twilio + STT + LLM + TTS. Move to custom only if you hit
Vapi's limits.

**Components:**

1. **Trigger:** form submission with "OK to call me" checked → Phase 2 router
   queues a `voice_callout` job
2. **Pre-call prep:** AI generates the call brief, voice agent loads it as
   system prompt context
3. **Call orchestration:** Vapi handles dial-out via Twilio AU number. AI agent
   identifies itself ("Hi, this is an AI assistant from UnwindAI..."), confirms
   they have 5 min, walks the script
4. **Tools the voice agent can call mid-conversation:**
   - `lookup_veec_count(activity, system_kw, zone)` — pulls from a static
     pricing table you maintain
   - `get_current_veec_price()` — pulls from a daily-refreshed cache of the
     ESC market price (~$54-$70 / VEEC at time of writing, lock to a
     conservative number for quotes)
   - `calculate_copay(system_cost, veec_count, veec_price)` — pure arithmetic
   - `book_followup(reason, when)` — schedules a callback with you
   - `transfer_to_human()` — bail-out: rings your mobile, joins the call
5. **Post-call:** transcript + outcome saved to Sheets/DuckDB. AI summarises
   into a 3-bullet brief for you. Lead status updated

**Compliance, non-negotiable:**

- AI identity disclosure within first 5 seconds
- Recording disclosure ("This call is being recorded for quality")
- Easy opt-out at any moment ("If you'd rather not continue, just say stop")
- Australian DNC register check (form opt-in already covers this — they invited
  the contact)
- Business hours only: 9am-6pm Mon-Fri local time
- Privacy Act compliance: data retention policy documented, opt-out honoured
  across systems
- NDS / Energy retail disclosure rules for the VEU activity tier you're
  operating under

**Pricing accuracy guardrail:**

The voice agent **must not invent numbers**. Two layers:

1. System prompt explicitly forbids inventing VEEC counts or prices
2. Pricing tools return data from a pinned table. If a question lands outside
   the table, the agent says "I'd need to confirm that with our team —
   can I get Thomas to call you back this afternoon with that number?" and
   triggers `book_followup`

**Voice quality target:** sub-800ms latency, ElevenLabs voice that doesn't
trigger uncanny-valley, natural barge-in (caller can interrupt mid-sentence
and the agent stops talking).

**Cost model:** 5-min average call ≈ $0.50-$1.50 in API + $0.07 in Twilio AU.
At 10 calls/day = $20-30/day operational. Worth it once close rate × deal
size justifies the spend.

**Done when:** a real Victorian homeowner who filled in the form gets a call
within 1 hour, walks through their options, and either books an in-person
assessment or gracefully opts out — without you ever picking up the phone.

### Phase 6: Knowledge base & content engine

**The pattern:** Phase 2-5 all consume from a shared knowledge base. That
knowledge base needs to be maintained.

**Content to maintain:**
- VEEC counts per activity per climate zone (Victoria has multiple zones)
- VEEC market price (daily, auto-refresh)
- System-specific data: capacity, COP, warranty, list price, install cost
- Eligibility rules (changes when ESC updates the program)
- FAQ patterns from past calls/emails

**Where it lives:** structured markdown in `context/` (already used by
`/api/chat`), supplemented by a DuckDB pricing table that the voice agent
tools query. Source-of-truth is the markdown — DuckDB is a fast lookup cache
rebuilt from markdown on deploy.

**Update workflow:** you edit the markdown when ESC updates the program. CI
hook regenerates the DuckDB cache. New voice agents pick up the new data on
next call.

### Phase 7: Lead admin & analytics

A small dashboard at `/admin/leads`:

- All leads in one place, filterable by source / category / status
- Click a lead → see full intake, call transcript, AI brief, drafted replies
- Inline status updates (won / lost / quoted / no-show)
- Conversion funnel: form → call booked → call held → quote sent → won
- Cost per lead, cost per win — close the loop on the AI spend

**Stack:** another static HTML page + a `/api/leads` endpoint reading from
DuckDB. Auth via magic-link email (Resend), not a username/password system.

## Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| RESEND_API_KEY not set on Vercel | High | Documented in WORKPLAN.md and ROADMAP, error message tells visitor to email directly |
| Claude API outage breaks chat | Low | Chat is non-critical — site still sells. Add status banner if it happens often |
| AI hallucinated VEU pricing | Medium | Phase 5 tool returns from pinned table, not model knowledge. Out-of-table → human callback |
| Spam Act / DNC violation | Low | Form opt-in covers it. Voice phase has explicit consent + disclosure |
| Lead drops in funnel between phases | Medium | Always email first as fallback. AI enrichment is bonus, not primary path |
| Vercel function cold starts | Low | Functions are small, cold starts <1s. Acceptable |

## Decision log (why we did things)

- **No SSR / no framework:** Karpathy minimum-code principle. Static HTML + JS
  fits the team size (1) and the actual product complexity. Adding Next.js
  would 3x the maintenance surface for zero user-facing benefit
- **Resend over SendGrid/Mailgun:** simpler API, better dev UX, fine pricing
  at this scale
- **DuckDB over Postgres:** Thomas's stack default, fits embedded model. If
  multi-writer becomes an issue, Phase 6+ can move to Turso or Postgres
- **Email-as-safety-net pattern (Phase 2+):** AI is bonus, not primary. We
  *never* lose a lead because an AI call timed out
- **Voice agent in Phase 5, not Phase 2:** first impressions are too valuable
  to gamble on day-one AI voice quality
